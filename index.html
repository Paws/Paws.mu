Such freedom to “return” from any location in your code to any other (within reason) is one of the core precepts of Paws. These powerful methodologies also allow for far more interesting and exiting API designs than most programmers are acclimated to using nowadays. Gone are the days of single caller, single set of ‘arguments,’ single return “methods.” With Paws’ inherent asynchronicity, API designers are free to lay out far more intuitive interfaces; interfaces that take multiple parameters from multiple callers, as they please (and whenever they please!); interfaces that provide data to multiple others at multiple times. More intuitive APIs means less to memorize, which (in turn) means easier and more natural programming!

Concurrent

Drop everything you think you know about concurrency, because in Paws, you get concurrency for free. We have reduced all of the normal concerns involved with writing concurrent code to a single mechanic for you to worry about; and as long as you carefully apply that mechanic across the entirety of your APIs, your code will be fully parallelized without any additional input from yourself. This mechanic is “ownership.” It can be thought of as a ‘data-side lock,’ of sorts; you simply declare ownership of any data you need either

  1. need multiple, sequential access to
    2. or
  2. expect to mutate
. In addition to managing the ownership of your calls, you must maintain the ownership-relationships of your data via responsibility, but that’s trivial! Using this information that you’re providing to the runtime, Paws can fully parallelize your code’s operations, while simultaneously preventing race conditions or other side-effects of unsafe concurrency.

Throughout the design process, we have continuously removed features and concepts from this core language, and instead re-implemented them in the language itself. The result of this process is a language that is, primarily, implemented in itself. This fact has one extremely important consequence for programmers like you: you can change almost any part of the language semantics, without modifying the interpreter! You (and your libraries) have just as much power over the language itself as we, the original designers of the language, did! However, we’ve taken this a step further yet; because not only are most of Paws’ semantics defined from within Paws itself … the syntax is as well! With our original, innovative, and slightly insane “interpretive preprocessor,” we bootstrap all of Paws’ syntactic elements out of an extremely basic set of core syntactics
those being the ‘cPaws’ (or “canonical Paws”) format mentioned above
. Again, this means that you have the same power of the language, including the syntax! Put succinctly: because Paws is dynamic enough to implement Paws in, it’s also dynamic enough to implement your own language in. It’s the intended end-game of Paws for each and every developer to have customized the language to their own tastes; everything from the object system to the parameterization and calling conventions can be completely changed to suit your tastes, and yet your code can still interoperate with the rest of the ecosystem.
